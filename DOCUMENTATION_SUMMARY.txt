================================================================================
SYSTEM OVERVIEW
================================================================================

The Vehicle Monitoring System has FOUR distinct user roles:

1. ADMIN (Role ID: 'admin')
   - Panel: /admin (Blue color)
   - Responsibility: System oversight and user management
   - Access: Complete system data and all Filament resources

2. INSTITUTE AUTHORITY (Role ID: 'institute_authority')
   - Panel: /authority (Purple color)
   - Responsibility: Final approval of vehicle registrations
   - Access: Registrations only, with approve/reject/sticker issuance actions

3. GUARD (Role ID: 'guard')
   - Panel: /guard (Orange color)
   - Responsibility: Vehicle access control at gates
   - Access: QR code scanning, plate lookup, own scan history

4. STUDENT (Role ID: 'student')
   - Panel: /student (Green color)
   - Responsibility: Vehicle registration and management
   - Access: Own vehicles and registrations only

================================================================================
KEY WORKFLOWS
================================================================================

STUDENT VEHICLE REGISTRATION WORKFLOW:
1. Student creates vehicle profile (vehicle details + document upload)
2. Student submits registration for approval
3. Admin optionally verifies the vehicle details
4. Authority reviews and approves or rejects
5. If approved: Digital sticker (QR code) is generated
6. Student downloads sticker and shows to guard
7. Guard scans at gate → Access granted/denied logged
8. After 1 year: Student requests renewal for new sticker

GUARD ACCESS CONTROL WORKFLOW:
1. Guard scans QR code or enters plate number
2. System validates sticker (status='valid' & within date range)
3. System displays vehicle/student info
4. Guard sees ACCESS GRANTED or ACCESS DENIED
5. System logs all scan data (who, what, when, result, method)

AUTHORITY APPROVAL WORKFLOW:
1. Authority views registrations in verified status
2. Authority sets sticker validity dates
3. Authority approves → DigitalSticker created with QR code
4. Authority can also reject with reason
5. Student receives notification

================================================================================
CRITICAL FEATURES
================================================================================

ROLE-BASED ACCESS CONTROL:
- User.canAccessPanel(Panel) enforces role-based access
- Each role has separate login page
- Only is_active=true users can login
- Query filtering prevents data leakage across roles/students/guards

DIGITAL STICKER SYSTEM:
- UUID-based QR code tokens (unique per sticker)
- Three statuses: valid, expired, revoked
- Automatic expiration based on validity_end_date
- Can be revoked by admin anytime
- Guards scan to verify access

REGISTRATION STATUSES:
- pending: Submitted by student, awaiting verification
- verified: Admin verified, awaiting authority approval
- approved: Authority approved, sticker issued
- rejected: Rejected by authority, student can request renewal

DATA SECURITY:
- Admin: No data filtering (sees all)
- Authority: Only sees verified/approved/rejected registrations
- Guard: Only sees own check-in logs (filtered by guard_id)
- Student: Only sees own vehicles/registrations (filtered by student_id)

================================================================================
DATABASE STRUCTURE
================================================================================

CORE TABLES:
- users: Base user accounts with role field
- students: Student profiles linked to users
- vehicles: Vehicles owned by students
- registrations: Registration requests with approval status
- digital_stickers: Generated QR codes for approved registrations
- check_in_logs: Guard scan records
- vehicle_types: Categories of vehicles (Car, Bike, Truck, etc.)

KEY RELATIONSHIPS:
User → Student (1:1)
Student → Vehicle (1:M)
Vehicle → Registration (1:M)
Registration → DigitalSticker (1:1)
DigitalSticker → CheckInLog (1:M)
Guard User → CheckInLog (1:M, via guard_id)

QUICK REFERENCE: WHO CAN DO WHAT
================================================================================

CREATE RECORDS:
- Admin: All records
- Authority: Digital stickers (via approval action)
- Guard: Check-in logs (automatic)
- Student: Own vehicles, own registrations

READ RECORDS:
- Admin: All records
- Authority: Only verified/approved/rejected registrations
- Guard: Only own check-in logs
- Student: Only own vehicles and registrations

UPDATE RECORDS:
- Admin: All records via forms or actions
- Authority: Registration approval/rejection/sticker issuance
- Guard: None (logs are read-only)
- Student: Own vehicles only

DELETE RECORDS:
- Admin: All records
- Authority: None
- Guard: None
- Student: Own vehicles only (not approved registrations)

================================================================================
COMMON QUESTIONS ANSWERED
================================================================================

Q: How does access control work at the gate?
A: Guard scans QR code → System validates sticker → Shows vehicle/student 
   info → Logs the scan → Notifies guard of access decision. Plate lookup 
   also available if QR scanner fails.

Q: What happens when a sticker expires?
A: Status automatically changes to 'expired'. Student sees it in their 
   registrations and can click "Request Renewal" to submit new registration 
   for approval.

Q: Can a student see other students' data?
A: No. All queries are filtered by student_id. The system returns no results 
   if user tries to access other students' data.

Q: What's the role of Admin vs Authority?
A: Admin manages system-wide configuration and can verify registrations 
   (optional). Authority is the final approver who issues stickers with 
   validity dates.

Q: Can a Guard see who scanned the QR code for a specific vehicle?
A: No. Guards can only see their own scan history, not other guards' scans.

Q: What happens if a sticker is revoked?
A: Admin clicks "Revoke" on the sticker. Status changes to 'revoked'. 
   Next guard scan with that QR will show "Access Denied: Sticker revoked".

Q: How are stickers generated?
A: Authority (or Admin) clicks "Approve & Issue Sticker" during approval. 
   They set validity dates (default: today to +1 year). QRCodeService 
   creates UUID token and PNG image. Student can download and share.

TESTING SCENARIOS
================================================================================

1. NEW STUDENT REGISTRATION
   - Student signs up and logs in
   - Creates vehicle with all details
   - Submits registration
   - Admin verifies
   - Authority approves with sticker dates
   - Verify student can download QR
   - Guard scans at gate → Access granted

2. EXPIRED STICKER RENEWAL
   - Wait for sticker to expire (or manually test)
   - Student clicks "Request Renewal"
   - New registration created (pending)
   - Authority approves
   - New sticker issued
   - Guard scans new QR → Access granted

3. ACCESS DENIAL SCENARIOS
   - Guard scans revoked sticker → Access denied logged
   - Guard enters plate with no sticker → Access denied logged
   - Guard scans expired sticker → Access denied logged
   - Guard scans valid sticker → Access granted logged

4. DATA ISOLATION
   - Login as Student1, verify can't see Student2's data
   - Login as Guard1, verify can't see Guard2's scans
   - Verify Authority can't access user/vehicle management

5. ROLE ACCESS CONTROL
   - Verify each role can only access their panel
   - Verify is_active=false users can't login
   - Verify role mismatch prevents panel access

================================================================================
MAINTENANCE NOTES
================================================================================

- Digital stickers use UUID tokens generated automatically
- QR codes are PNG images stored in storage/public/
- Check-in logs are immutable (no delete, only view)
- Registrations can be in 4 states - transitions are one-way (mostly)
- Student data is cascade-deleted when student profile is deleted
- Guards should have unique IP tracking for auditing

================================================================================
ARCHITECTURE STRENGTHS
================================================================================

1. Role Isolation: Each role has minimal access to what they need
2. Data Security: Query filtering at model level prevents data leakage
3. Approval Workflow: Multi-step verification (student → admin → authority)
4. Access Logging: Every scan is logged for auditing
5. Flexible Access: QR code OR plate number scanning options
6. Renewal System: Automatic handling of sticker expiration
7. Filament Integration: Clean, modern admin interface
8. Status Tracking: Audit trail with verified_by, approved_by, timestamps
